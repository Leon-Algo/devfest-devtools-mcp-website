AI 控制浏览器：DevTools 不再只是工具栏

AI 通过 DevTools MCP Server 直接调用 Chrome DevTools，自动执行你原本要亲自完成的调试工作。

“浏览器从单机调试工具，变成可以被 AI 编排的远程控制节点。”

AI 能帮你做什么？

  • 自动跑 Lighthouse / 性能 Trace，提取 LCP / CLS 关键信息。
  • 监听 Console，第一时间捕捉并解释错误堆栈。
  • 分析 DOM 结构，生成结构化文档或可视化快照。
  • 批量截图、录制交互过程，用于复现和报告。

MCP：AI 与工具世界之间的通用插口

一次接入，多种能力。DevTools MCP Server 只是其中一个端口。

1 · 定义“插口”

MCP 用统一的协议描述“工具有哪些能力、怎么调用、如何返回结果”。AI 不再跟 HTTP 细节打交道，而是和一套稳定的 schema 对话。

2 · 一次接入，多处复用

连接 DevTools 之后，你可以用同一套 MCP 框架接 GitHub、Issue 系统、监控平台……AI 能在同一个“上下文空间”协同这些工具。

3 · 形成“工具编排层”

在 Workshop 中，我们会从 DevTools 出发，走完整一条链路：调试本地页面 → 提交 GitHub → 触发 Actions → 部署到 Pages → 再由 AI 对线上站点体检。

核心隐喻 MCP 像是什么？

你可以把 MCP 想象成机房里的配线架： 所有线缆都整理到统一入口，然后再分发到不同设备。 AI 看到的是“端口编号”和“能力说明”，而不是乱成一团的网线。

一条从命令行到浏览器，再到线上生产的路径

Workshop 不是一场 PPT，而是一条实打实跑完的工程链。

01

本地 HTML · 问题现场

从一个真实存在问题的本地页面开始：加载异常、Console 报错、布局不稳定。

02

DevTools MCP · 调试控制室

通过 DevTools MCP，AI 接管浏览器：打开页面、抓 Performance、记录 Console、拍快照。

03

GitHub · Actions · Pages

把修好的页面推送到 GitHub 仓库，用 Actions 自动部署到 GitHub Pages。

04

AI 体检 · 线上站点

再让 AI 访问线上 URL，对性能和错误进行一次“验收测试”，形成完整闭环。
